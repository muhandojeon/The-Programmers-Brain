# 신속한 코드 분석

- 연구에 의하면 프로그래머는 60%의 시간 동안 코드를 이해하는데에 사용함

> 쓰려고 이해하니까 뭐 .. 그럴 수 있겠다 싶음

## 코드를 신속하게 읽기

- 프로그램은 사람이 읽을 수 있도록 작성해야만 한다. 기계가 실행하는 것은 부차적인 일이다

- 패턴을 청크화 하면 빠르게 읽고 이해할 수 있다
  - 그래서 LTM에 지식이 많으면 기억을 쉽게 한다는 사실은 프로그래밍에도 해당 (ex. 체스)

- 마법의 수 7 플러스 마이너스 2: 인간의 정보처리 용량 제한

## 읽는 것보다 보는 것이 더 많다

- 무언가 읽을 때 먼저 영상 기억 공간에 저장함

- 어떻게 하면 읽기 쉬운 코드를 작성할 수 있을까?
  - 디자인 패턴을 이용하면 된다

  - 청크로 나누기에 도움이 되는 주석문을 달자
    - 저수준 주석문은 오히려 청킹에 부담이다

> 위의 경우 아니면
> 이유, 의도, 맥락이 모호한 부분에 주석을 다는 것을 좋아하긴 함

- 50줄 짜리 코드 2분 보고 외워서 작성해보는 훈련
  > '내 수준을 알기 위해' 보다는 그냥 한 번 해보면 재밌을 거 같았음

## 요약

- 정보를 기억할 때 STM은 크기에 대한 제약을 극복하기 위해 LTM과 협업
- 새로운 정보를 읽을 때 우리 두뇌는 청크라는 몇 개의 묶음으로 나눔
- 코드는 우리 두뇌에서 처리하기 쉽게 만드는 특징들. 가령 디자인 패턴, 주석문, 명확한 표식 같은 것들을 가질 수 있다.

> 결국 읽기 좋은 코드를 작성하는 초석을 만들기 위해서
> 많이 읽어서 정규화된 패턴들을 학습하는 게 중요하다? 정도로 느껴졌는데
>
> 요즘에는 코드를 읽고 이해하려는 노력보다 '/sc:explain' 으로 딸깍 하는 모습이 스스로 자주 나오지 않나 싶음
> 그래서 곡선이 완만해진 느낌인가 싶기도
