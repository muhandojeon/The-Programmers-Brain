# 2장 신속한 코드 분석

연구에 의하면 프로그래머의 시간 중 거의 60%를 코드를 작성하는 게 아니라 이해하는 데에 사용한다고 함

따라서 정확도를 유지하면서 코드를 빨리 이해하도록 향상한다면 프로그래밍 기술이 크게 개선되는 셈

## 자바 코드를 기억하기 위한 실험 내용

의도적으로 for 루프의 변수명을 b나 l처럼 잘 쓰지 않는 생소한 이름을 사용하면 패턴을 찾고 기억하는 것이 어려워짐

=> 왜 이 코드는 기억하기가 어려웠을까? 가장 결정적인 이유는 STM의 용량이 제한되어 있고 저장 시간은 30초를 넘기지 않기 때문 ㄷ ㄷ

좀 더 최근의 연구에서는 STM의 용량이 2개~6개 사이로, 심지어는 더 적다고도 추정한다고 함

그럼 기억의 크기 제한을 극복하기 위해서는??

### 단위로 묶는 위력

실험 1 

전문 체스 플레이어가 보통 체스 플레이어보다 체스판 위에 임의로 배치한 말의 위치를 몇초간 외우도록 블라인드 테스트 했는데 훨씬 더 잘 기억했다고 함

근데 이건 말의 위치를 기억하는 방법 중에 시실리언 오프닝(처음에 두는 수)를 이용하거나 이전 경기를 통해 기억했었기 때문에 (LTM을 통해 기억했기 때문에) 더 잘 기억할 수 있었고

덕분에 나머지 말의 위치를 STM에서 불러오는데 더 효율적으로 사용했음 (왜냐면 '시실리언 오프닝'을 하나의 청크로 생각하면 STM의 기억 공간 하나만을 차지하니까 ㄷ ㄷ)

다른 케이스로 실제 체스 게임에서는 일어날 수 없는 위치에 무작위로 놓고 블라인드 테스트 하면 두 플레이어 모두 기억 잘 못한다고 함

실험 2

문자 12개를 5초간 외워야 된다 
- 낯선 문자 조합(O|3 ,.C(* l|3c)
- 조합 안되는 알파벳 문자 (abk mrtpi abqr)
- 영단어 (cat loves cake)

마지막 영단어는 여러개의 문자를 청크로 묶어서 3개만 생각하면 됨. STM의 최대 용량에 훨씬 못 미쳐서 쉬운데 앞 두 예제는 어려움; STM의 한도를 넘어서기 떄문에;

그래서 LTM에 지식이 많으면 기억을 쉽게 한다는 사실은 프로그래밍에도 해당함

실제로 숙련된 프로그래머들이 초보 프로그래머들보다 코드를 잘 기억한다는 실험 결과도 냈음

## 기억하는 대상이 중요한 것이 아니고 기억하는 방식이 중요하다

알골 언어 키워드를 초,중,고급 프로그래머들한테 외워보라고 했을 때 초급 프로그래머들은 기억하는데 도움되도록 문장을 만들어 사용했는데 전문가 행님들은 키워드를 그룹 지을 때 이미 가지고 있는 지식을 사용했음

앞선 실험들을 통해 일상적이고 예상 가능한 상황은 청크로 묶는 것을 쉽게 한다는 것을 알게 되었는데, 그룹으로 나누기 쉬운 코드를 작성하는 방법엔 뭐가 있을까?

### 디자인 패턴 사용

### 주석문 쓰기

코드에 주석문이 있으면 새로운 팀원이 코드를 쉽게 이해할 수 있음

> AI agent 한테 코드 작성 시키면 주석을 달아주는데, 이걸 우리 조직의 톤앤 매너에 맞춰서 수정하는 작업을 꼭 해주고 있음

'이 함수는 주어진 이진트리를 중위 순회하며 프린트 한다' 같은 고수준 주석문은 코드를 청크 단위로 쪼개는 데 도움이 됨

반면, i++;에 'i를 1만큼 증가' 같은 저수준 주석문을 넣는 것은 오히려 청킹 작업에 부담이 된다고 함

> 이거 진짜 인정. AI agent가 가끔 단순 코드에도 주석을 달아서 정리 작업 해줘야 됨

### 표식 남기기

표식은 일반적으로 코드 내에 사용하는 특정 알고리즘, 자료구조 혹은 접근 방식등을 보여주는 라인을 뜻함

예를 들어 파이썬 코드에서 이진트리가 자료구조로 사용된다는 사실을 이 표식으로 부터 유추할 수 있음

- 주석문에 '트리'라는 단어 사용
- root와 tree라는 변수명
- left, right 필드명
- 트리에 관한 문자열 값

표식은 코드를 읽고 이해하는 과정에서 소스 코드에 대해 개발자가 갖는 가정이 맞거나 틀린 것을 확인해주는 역할을 수행하기 때문에 매우 중요함
