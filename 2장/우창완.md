### 2.1 코드를 신속하게 읽기

프로그램은 사람이 읽을 수 있도록 작성해야만 한다.
=> 코드 자체는 STM이고, 코드에 대한 문맥이 LTM이 저장되기 때문이지 않을까?

#### 생소한 코드를 읽는 것은 왜 어려운가?

작업할 때 STM의 양은 한정되어있고, STM은 LTM보다 훨씬 더 작기 때문
STM은 2~6개 사이라고 한다.(개수는 어떻게 측정?, 최대 기억의 단위가 6개라는 걸까?)

### 2.2 기억의 크기 제한을 극복하기

위에서 살펴봤듯, 한정된 STM의 용량을 극복하려면 문맥을 chunking해야한다.

사례:
체스기사는 자신이 경기에서 둔 모든 수를 기억할 수 있다.
하지만, 아무런 맥락없이 두어진 수를 본다면 일반인과 큰 기억력의 차이가 없다.

체스도 당연히 문맥이 있다. 문맥을 알고 있다면 STM대신 LTM에서 인출해오는 효과가 있지 않을까?
코딩에 대입해보면 코드도 당연히 맥락이 있다. 맥락에 맞는 코드와 구조가 인지부하를 줄이면서 코드를 이해하고, 또 작업할 수 있을 것 같다.

### 2.3 읽는 것보다 보는 것이 더 많다.

정보는 STM에 도달하기 전에 감각 기억 공간(특히, 영상 기억 공간)을 거친다.

영상 기억공간(글을 읽을 때, 단락별로 읽는 느낌이지 않을까?)이 있는데, 이는 STM보다 더 용량이 크다.

#### 2.3.2 기억하는 대상보다 기억하는 방식이 중요하다.

어떠한 글, 문맥을 읽을 때 개인마다 모두 다른 background로 글을 읽는다.

동일한 영어 단어를 외우더라도 서로 다른 그룹으로 연관지어(chunking) 기억한다.

개발을 모르는 사람: TRUE IS REAL THEN FALSE
개발을 아는 사람: TRUE FALSE IF THEN ELSE

코드가 일정 패턴을 따라 구현되어 있다면, 패턴을 기반으로 이를 chunking하여 기억, 작업한다.
패턴화된 코드베이스를 수정하는 시간이 그렇지 않은 코드를 수정하는 시간보다 짧았음.
